# ManabiGrid 制作プロトコル（公開版）

このドキュメントは、ManabiGrid の単元教材を作るための工程契約です。外部の貢献者が**Claude Fable 5**でこの工程を再現できるように書いてあります。プロンプトの実物は [prompts/](prompts/) にあります。

> **使用モデルについて**: 新単元の制作に使うAIは **Claude Fable 5**（reasoning effort medium以上・サブエージェントも Fable 5）に限定しています。理由と申告方法は [CONTRIBUTING.md](../CONTRIBUTING.md) を参照。誤り報告・改善・デザインの貢献はモデル不問です。

## 0. 全体像

単元は1回の依頼で書かせず、**工程ごとに別々の会話（新規チャット）で**AIに実行させます。調べる者・要約する者・書く者・検査する者を分離して、思い込みや誤読の混入を構造的に減らすためです。

```text
[工程1] 調査
  → [工程2] 正規化（調査ノート → 執筆用サマリー）※最重要リスク工程
  → [工程2b] 独立ミニ監査（高リスク記述の原文照合・対象単元のみ必須）
  → [工程3] 執筆（執筆用サマリーのみを入力）
  → [工程4] セルフQA（執筆と別の会話で）
  → [工程5] 別系統AIによる批判レビュー（メンテナ側で実施）
  → [工程6] 人間レビュー（メンテナ側で実施）
```

原則:

- **各工程は別コンテキストで起動する**。長い一続きの会話で作ると後半で品質が下がるため、工程間の受け渡しは定義された成果物ファイルのみとし、会話の文脈を持ち越さない。
- 作る者と検査する者を分ける。検査担当は修正せず、指摘だけを返す。修正は執筆側が行う。
- 未確認のことを確認済みに見せない。分からないことは verify_required として正直に残す（残っていて構わない——残っていることが正直さの証明）。

## 1. 各工程の契約

### 工程1: 調査

- 入口条件: 対象単元の確定（校種・学年/科目・領域）
- 入力: 調査発注テンプレート（[prompts/research_order_v2.2.md](prompts/research_order_v2.2.md)）の変数を差し替えたもの
- 禁止: 教材本文の執筆／記憶からの数値記載／書誌情報だけ確認したものを「確認済み」扱いにすること
- 成果物: 調査ノート（research_notes.md）——3分離（確認済み／推論／未確認）＋出典一覧（source_list）＋たどり着けなかった資料の表＋残った未知の列挙
- 品質規則:
  - 一次資料主義: 学習指導要領（平成29/30年告示）本文・解説の該当箇所を実際に取得して読む。数値・文言はすべて「資料名＋年度＋URL＋確認日」付きの転記値。
  - 原典に届かず孫引きしか確認できなかったものは「引用の引用」と明記する（確認済みにしない）。
  - 転載禁止: 市販教科書・過去問・調査問題の本文/図/数値設定を写さない（出題傾向・誤答の類型を自分の言葉でまとめるのは可）。
- 失敗時: たどり着けなかった資料は表に記録して続行する（主要資料が全滅した場合のみ人間へ相談）。

### 工程2: 正規化（**最重要リスク工程**）

この工程が本プロトコルの**単一故障点**です。ここで意味が変わると、工程3（サマリーしか見ない）・工程4（サマリー基準で検査）のいずれも構造的に検出できません。したがって検収を最も厚くします。

- 入口条件: 工程1成果物の存在
- 入力: 調査ノートのみ（**Webでの再調査はしない**。抜けを見つけたら工程1へ差し戻すか「未確認」と記載する）
- 禁止: ノートにない事実の追加／確認度（confidence）の根拠なき格上げ／原典分類語の変換（下記6項目）
- 成果物: 執筆用サマリー（writing_summary.md）＋**高リスク記述台帳**

**赤信号語彙**（これを含む記述は自動的に高リスク扱いとし、台帳に載せる）:

> 正答／準正答／誤答／無答／反応率／解答類型／期待される解答／例／ただし／除く／〜ではない／部分点／評価しない／条件を満たさない

**高リスク記述台帳**（赤信号語彙を含む記述のみ。全記述には課さない）:

| claim_id | 原文の場所 | 原文 | 正規化後 | 変換の種類 | 意味変化の懸念 |
|---|---|---|---|---|---|

- 変換の種類: 転記そのまま／要約／言い換え／統合 のいずれか
- 意味変化の懸念: あれば内容を書く（なければ none）

**通過条件「原典分類語の保持」6項目**（1つでも違反があれば工程2やり直し）:

1. 「準正答」を「正答」に変えていない
2. 誤答傾向を「〜できない」という断定に変えていない
3. 反応率・正答率の数値を「多くの生徒が」等のあいまいな頻度表現に変えていない
4. 年度・設問番号・適用条件を落としていない
5. 例示を「代表例」「すべての場合」扱いにしていない
6. ただし書き・除外条件・否定形（〜ではない）を落としていない

**執筆用サマリーの必須スロット「誤読禁止メモ」**: 「XをYと解釈しないこと」の形で列挙する（例:「反応率42%を『半数近くができない』と解釈しない」「準正答例を模範解答と解釈しない」）。高リスク記述がある単元では空にできない。

検収は「サマリー内部の自己整合チェック」ではなく「**原文→サマリーの意味保存検査**」で行う。高リスク記述ごとに次の5点を突き合わせる:

1. 調査ノートの原文（表を含む該当箇所そのもの）
2. サマリー本文の対応記述
3. 執筆者に渡る最終記述
4. 確認度（格上げされていないか）
5. 執筆注意（ただし書き・条件がサマリーまで生きているか）

### 工程2b: 独立ミニ監査（別の会話で）

- 目的: 工程2の誤読を検出する唯一の構造的な検出点。工程2とは**別の会話**で起動する
- 入口条件: 工程2成果物（サマリー＋高リスク記述台帳）の存在
- 入力: 調査ノート＋高リスク記述台帳（**再調査はしない**。Web接続不要）
- 作業: **高リスク記述のみ**を調査ノートの原文と照合し、意味変化を赤信号として返す。**修正は禁止**（赤信号の返却のみ。直すのは工程2側）
- 成果物: ミニ監査結果（claim_idごとに OK／赤信号＋根拠となる原文箇所）
- 適用範囲: 最終候補（公開候補）単元および専門レビュー必須領域（§4）に該当する単元は必須。それ以外は省略可（省略した場合は来歴情報に縮退フラグを残す。§6参照）

### 工程3: 執筆

- 入口条件: 執筆用サマリーの完成（ミニ監査対象単元は監査通過後）
- 入力: **執筆用サマリーのみ**（調査ノート全文・調査ログは渡さない＝ノイズ遮断）
- 禁止: サマリー外の事実断定／解答の本文混在／対象読者レベル外の内容（入試技巧・先取り）を本編に混ぜること
- 成果物: lesson_map（単元設計）＋レッスン本文＋解答別ファイル＋出典リスト。本文は**タグ付きcore形式**（§2参照）
- 品質規則: 執筆契約（[prompts/writing_contract_v2.5.md](prompts/writing_contract_v2.5.md)）に従う。要点は §2〜§3。
- 検証: 工程4に委ねる（自己検証で完了扱いにしない）

### 工程4: セルフQA（執筆と別の会話で）

- 入口条件: 工程3成果物の存在
- 入力: 教材ドラフト一式＋執筆用サマリー＋QAチェックリスト（[prompts/qa_checklist_v1.md](prompts/qa_checklist_v1.md)）
- 点検項目: チェックリストの①〜⑪（数値の年度なし使用・サマリー外断定・解答漏れ・レベル外混入・範囲外・数値解答の再計算・他者前提活動・指標名照合と合計再計算・文体点検・中心例の意味検算・量化/定義域/包含の検査）
- 成果物: QA結果（指摘は「箇所・違反した規則・修正案」の3点セット。指摘がなければ「なし」と明言）
- 失敗時: 工程3へ差し戻す（**検査担当が直接修正しない**＝作る者と検査する者の分離）

### 工程5: 別系統AIによる批判レビュー（メンテナ側）

同じ系統のAIで工程1〜4を回しても、**そのモデル自身の共有盲点は検出できません**。そこで最終候補単元には、制作に使ったのとは別系統のAIによる批判専任レビューを掛けます。

- 入口条件: 工程4通過
- 入力: 執筆用サマリー＋教材ドラフト＋高リスク記述台帳（重点確認対象として明示。調査ノート原文は渡さない）
- レビュアーへの依頼内容: 調査や書き直しはさせず、①事実の誤読・誤解釈の疑い ②重要な抜け（扱うべきなのに欠けている観点・つまずき）③出典の弱い断定 ④生徒に誤解を与えうる表現——の4点だけを、各指摘に確信度（高/中/低）付きで報告させる
- 雑談枠の事実確認: 教材が実在の逸話・人物・出来事を雑談枠で使っている場合、その逸話の事実確認（事実性・通説/俗説の区別）をレビュー項目に含める。事実確認が済んでいない実在逸話は使用不可
- 指摘の扱い: 鵜呑みにせず、一次資料に戻って裁定する。裁定は判断記録として残す（§6）

### 工程6: 人間レビュー（メンテナ側）

人間の全文レビューではなく、**赤信号項目に限定した検収**を行います（疲労による「問題なさそうで通す」事故を防ぐため。それ以外は各工程の機械的チェックに委ねる）:

- 調査の検収: 出典一覧の欠落・到達不能資料の全滅・「引用の引用」の確認済み混入・確認の深さ（本文まで/抄録のみ/書誌のみ）の未記載、のみ
- 正規化の検収: 高リスク記述台帳の懸念フラグ付き項目・ミニ監査の赤信号・原典分類語6項目の違反疑い、のみ
- QAの検収: サマリー外断定の指摘・再計算不一致・解答漏れ・レベル外混入の指摘、のみ
- 批判レビューの裁定: 確信度「高」の指摘は必ず裁定（中・低はまとめて）

### 未レビュー成果物の扱い

批判レビュー・人間レビューを通過していないドラフトが「ほぼ完成品」として扱われる事故を防ぐため、通過前のドラフトは frontmatter に未レビュー状態（`published_draft` 等）を明示し、「完成」報告には使いません。

**公開との関係（2026-07-16方針変更・BUILD_LOG参照）**: 未レビューであることを**明示した候補ドラフトとして**リポジトリへ同梱・公開することは、現行方針では認めます（READMEと各ファイルのfrontmatterで候補ドラフトであることを表示）。「人間レビュー済」等の最終状態への昇格は、通過・承認後にのみ行います。

## 2. タグ付きcore形式（本文の書き方）

本文は1回だけ書き、宛先の異なる部分をブロックタグで分けます（生徒用・ガイド用などの各ビューは、このタグ付きcoreから機械的に生成します）:

- `:::guide` — 自習者が読める「一段深い解説」層。裏方メモの文体にしない（自習者が読む前提で書く）
- `:::answer` — 解答・解説（ビュー生成で本文から分離される。本文に答えを直書きしない）
- `:::zatsudan` — 先生の雑談枠（各レッスン1箇所・2〜4文・節の切れ目に。実在の逸話は事実確認前に使わない。健康・災害・政治隣接の話題は禁止）
- `:::stretch` — 発展（上位層向け。標準本文と区別する）
- `:::plus` — 標準範囲外の追加コンテンツ（本編に混ぜない）

タグの外に書いたものが標準本文（全読者向け）になります。

## 3. 執筆の設計原則（要点）

詳細は執筆契約（[prompts/writing_contract_v2.5.md](prompts/writing_contract_v2.5.md)）を正とします。

### 3.1 学習者単独成立の原則

想定読者は**ひとりで学び直す独習者**であり、学校・教室での使用は設計上考慮しません。

1. すべての活動・練習・評価は**学習者一人で完結する形を標準**とする
2. 他者の協力を前提にした活動（クラスメイトへのインタビュー・ペア/班活動・発表・掲示・家族への協力依頼・先生への質問を必須とする設計）を本文に置かない
3. 許容する外部リソースは辞書・教科書・検索エンジン・AIチャット。検索やAIの利用を促す場合は、**検索フレーズ例やAI宛のプロンプト例を併記**する（丸投げ指示にしない）。AI活用枠は「独りでは代替しにくい」場面（会話相手・添削・記述の壁打ち等）に限定
4. 多人数版・教室版の運用は、必要なら `:::guide` 側の「教室で使う場合」オプションとしてのみ記載可（本文の標準手順にはしない）。実在の人を扱う置き換えをそこに書く場合は、本人同意・共有範囲・答えたくない項目への配慮条件を必ず併記する
5. **生徒が読む層には実在の人物（生徒・家族含む）の個人情報を扱う活動を設計しない**。AI・検索の利用例でも、実在の個人情報を入力させる例文を作らない（架空人物のみ）

### 3.2 文体規約

1. 呼びかけは「〜してみよう」を基本形とする（「〜してみて！」形は使わない）。親しみやすい語り口でよい——おカタくしない
2. 「。」の硬い連続を避け、「？」「！」を適切に使う（1セクション1箇所程度・誘う調子）
3. 罫線ダッシュ（——）はキメる時だけ・1レッスン1〜2回目安。AIっぽい文体要素を避ける: ——の多用／「そう、」「実は」の乱発／比喩の連打／不必要な体言止めの連続
4. 固有名詞・固有表現は初出時に読みを括弧書きで併記。重要な定義は「【ことば】」の定型枠（引用ブロック）で示す
5. 漢字は学年配当にとらわれず普通に使う。学年を超える漢字・難読語の初出には**ふりがなを括弧書きで併記**（例: 斜辺（しゃへん））。ふりがなは原稿データとして保持し、表示切替できる形を想定
6. 手書き・教科書表記と差異が出る記法（√・分数・指数の1行表記等）は、初出時に「ノートではこう書く」の橋渡しを置く
7. 印刷はマストではない——紙幅を理由に品質・説明の丁寧さを落とさない（ただし冗長化の免罪符にもしない）

### 3.3 媒体・層・導線

1. 画面・紙**完全等価**——媒体依存表現（クリック/タップ/ページをめくる等）を書かない。インタラクティブ要素には紙上の代替手順を併記する
2. 標準層は中間〜苦手層。上位向けは `:::stretch` に。発展への導線（調べるフレーズ・用語解説）を任意スロットとして置ける
3. 解答は別ファイル分離を維持し、レッスン末に解答ファイルへの明示リンクを置く
4. 英語は発音記号・カタカナ近似を本文併記する

### 3.4 図表方針

1. **各レッスン最低1枚を努力目安**とする（概念整理図・手順図等。装飾のためだけの図は作らない）
2. 図・表で説明できる箇所は、制作の手間が掛かっても図表を優先する（全教科）。長い文章説明の連続は図表で分節する（「目が滑る」対策）
3. 図は **「生成スクリプト→SVG」方式（コード来歴）** で作る。SVGの `<title>` `<desc>` に図の意図・主要数値・再現に必要な説明を埋め込み、巻末資料の来歴導線と対にする（後から誰でも検証・再生成できるように）
4. 数値例・データは自作する（実在の調査の数値設定を写さない）。作った数値は「教材用に作成した架空の値」と明示する

## 4. 専門レビュー必須領域

以下は赤信号項目限定の軽量検収では通さず、人間（必要なら外部専門知識の確認を含む）レビューを必須とします:

1. 理科: 実験・観察の安全に関わる記述
2. 社会: 憲法・統治機構、政治的評価を含む記述
3. 国語: 実在作品本文の扱い・著作権
4. 英語: 準正答の分類（正答/準正答の線引き）
5. 高校の細分化された科目内容（専門性が中学より高い）

## 5. 来歴の考え方（本プロジェクトが約束するもの）

公開時に約束するのは**品質ではなく来歴の記録**です。記録の中身:

1. **生成の来歴**: どの工程が、いつ、何を入力に生成したか（各成果物の frontmatter＋来歴ブロック）
2. **事実の来歴**: 主要な数値・文言について「原典確認/引用の引用」「取得方法」「確認日」を出典一覧で追跡可能に保つよう努める（引用と出典台帳の機械的な1対1対応（source_id方式）は未導入の改善課題——docs/RELEASE_RECORD_20260717.md「既知の制限」2参照）
3. **検証の来歴**: 何が verified（検証済み）で何が verify_required（要検証）か、どの工程・誰がいつ検証したか
4. **判断の来歴**: 矛盾の裁定・採否の判断は判断記録として残す
5. **免責の明示**: verify_required 項目の存在と、誤り報告→修正の経路を成果物に明記する

守るべきルール:

- **verify_required を勝手に外さない**。外せるのは、実際に原典を確認して検証した者が、確認日・確認方法を記録した場合だけ
- **来歴の事後改変の禁止**: 日付・確認状態を遡って書き換えない。**訂正は追記で行い、履歴を残す**
- verified / verify_required の区別を成果物から削って「全部確認済み」に見せない（区別が残っていることが信頼の根拠）

## 6. 縮退の記録

工程を省略した単元（例: ミニ監査の省略）は、来歴情報に縮退フラグを残します。縮退フラグ付き単元が増え続けたら、それは縮退ではなく仕様劣化とみなして立ち止まります。

## 7. 公開可否の5ゲート

「教材として良いか」と「公開してよいか」は**別の判断**です。公開は以下5ゲート全通過のみ（1つの総合判断で「大丈夫そう」と通しません）:

1. **教材妥当性**: セルフQA・批判レビュー・人間検収を通過しているか
2. **来歴追跡性**: 来歴ブロック・出典一覧・高リスク記述台帳が揃っているか
3. **権利**: 著作権・転載条件・出典表示（JP-COS LODはCC BY 4.0）のチェックが完了しているか
4. **内部資料の混入なし**: 調査ノート原文・内部メモが同梱されていないか
5. **再現可能性**: 生成条件（工程・テンプレートのバージョン・日付）が説明できるか

# METHODOLOGY — この教材はどう作られたか

本書は、本リポジトリの教材の制作方法・AIと人間の役割分担・検証の到達点を、事実に即して開示するものです。

## 1. 保証するもの・しないもの

この教材が保証するのは、**内容が正しいことそのものではなく、「どう作られ、何を根拠にし、どこまで検証されたか」を追跡できること**（来歴の記録と追跡可能性）です。

- 教材の正確性や、あなたの利用目的への適合性は保証しません。
- `verify_required` の標記がある項目は、授業・配布・販売・外部公開の前に、利用者側で原典を確認してください。
- 本プロジェクトは、文部科学省・国立教育政策研究所その他の出典元による公式・公認教材ではありません。
- 誤りを見つけたら報告してください（Issueテンプレートを用意しています）。訂正は履歴を残して行い、過去の確認状態を遡って書き換えることはしません。

なぜ「品質保証」でなく「来歴保証」なのか——AIが主工程を担う教材で「絶対に間違いがない」と約束することは誠実ではありません。代わりに、すべての事実・数値・判断について「どこから来たか」「どの工程がいつ検証したか」「何が未検証か」を機械的に追える状態を保ち、誤りが見つかったときに根から直せる構造を保証します。

## 2. AIと人間の役割分担

本教材は、**調査・執筆・QA（品質検査）・図版生成・体裁整備といった主工程をAIが担い、人間は工程の設計・各段階の裁定・最終レビューを担う**分担で制作されています。

- **AIの担当**: 一次資料の調査、執筆用要約への正規化、本文執筆、検査専任のQA、図版のスクリプト生成、タグ付き原稿からのビュー展開。
- **人間の担当**: 制作規約と工程の設計、レビュー指摘の採否の裁定、外部レビューの投入操作、実務者（塾講師）としての最終検収、公開可否の判断。
- 人間レビューが完了するまで、すべての成果物は候補ドラフト（candidate_draft）として扱い、最終版とは呼びません。

## 3. 制作工程（多段パイプライン）

1単元は次の多段工程を通ります。各段は**別々のコンテキスト（役割分離）** で実行され、成果物と裁定は記録に残します。

| 工程 | 内容 | 担当 |
|---|---|---|
| 調査 | 学習指導要領・同解説、全国学力・学習状況調査などの一次情報を収集。事実を「確認済み／推論／未確認」の3つに分けて記録 | AI |
| 正規化 | 調査記録を執筆用要約へ整理。誤引用しやすい数値・条件つき事実の台帳（高リスクclaim台帳）を作成 | AI |
| ミニ監査 | 要約と調査原記録を突き合わせ、正規化の過程で意味が歪んでいないかを別コンテキストで検査 | AI |
| 執筆 | 要約**のみ**を入力に本文を執筆（執筆中のWeb再調査禁止・要約外の事実断定禁止）。調査数値は生徒向け本文に書かず、指導者向けの設計根拠層に出典セットで記載 | AI |
| セルフQA | 検査専任（修正禁止・指摘のみ）。全計算のプログラムによる再計算、禁止事項の機械検査などの検査契約に基づく | AI（別コンテキスト） |
| 外部批判レビュー | **別ベンダの高性能AIに批判レビューをさせ、同一系統モデルの共有盲点を突く**。指摘の採否と理由を全件、裁定記録として残す | 別ベンダAI＋人間の裁定 |
| 改稿 | QA・外部レビューの指摘を裁定に従って反映し、修正内容と検証を記録 | AI |
| 図版 | 全図版をPythonスクリプトで生成（手描き・借用画像ゼロ）。生成時に本文数値との一致を自動検算し、解答の数値が図に漏れていないかも機械検査（詳細は SPEC_figures.md） | AI |
| タグ整備・ビュー生成 | 本文を役割タグつきの単一原稿（core）として整備し、生成器で複数のビューに展開（詳細は SPEC_tagged_core.md） | AI＋生成スクリプト |
| 人間レビュー | 実務者（塾講師）による検収。完了までは全成果物が候補ドラフト | 人間 |

**内部工程名との対応**: 制作記録・図版台帳・teacher_notes などの来歴記述には、制作時の内部工程名がそのまま残っている箇所があります。読み替えは次のとおりです——**パスA/B＝調査（＋正規化）**、**パスC＝執筆**、**パスD＝セルフQA**、**B2＝内部品質検査（セルフQA工程の内部名）**、**パスE＝外部批判レビュー**、**工程⑧＝人間レビュー**。

**執筆契約の版について（v2.2→v2.5の主な差分）**: 同梱単元の多くは執筆契約v2.2系（frontmatterの `style_version` 参照）で制作されており、公開している契約は最新のv2.5（[prompts/writing_contract_v2.5.md](prompts/writing_contract_v2.5.md)）です。v2.2からの主な追加は——v2.3: 学習者単独成立の原則（ペア活動・発表など他者前提の活動を書かない）／v2.4: 文体・表記規約（読み併記・文末調子などの明文化）／v2.5: 図表の積極採用と図版へのAI再利用メタ情報（SVGの `<title>`/`<desc>`）の埋め込み——で、いずれも品質規則の追加であり、v2.2系で作られた既存単元の本文を無効にする変更ではありません。厳密な再現を要する場合は版差がある点に留意してください。

## 4. 使用したAI（事実表示）

- **調査・執筆・QA・図版・タグ整備（教材本文の全主工程）**: **Claude Fable 5**（Anthropic・model id: claude-fable-5）。工程ごとにコンテキストを分離し、サブエージェントを使う場合も Fable 5 で統一。
- **外部批判レビュー**: **GPT-5.6 Pro**（OpenAI）。人間が手動で投入し、指摘の採否は全件裁定記録に残す。
- **工程設計の外部レビュー・権利面の反証レビュー**: 同じく GPT-5.6 Pro。
- **例外（正直な開示）**: ツールのうちビュー生成器（tools/view_generator）の初版実装は **OpenAI Codex（GPT-5.6系）** が行い、Fable 5 が独立検証しました。**教材本文そのものは、同梱済みの全単元が Fable 5 のみ**で執筆されています。なお、理科には Fable 5 の生命科学分野の安全機構により執筆できない単元があり、その単元に限って複数モデル分担の例外工程を定めています（[MODEL_POLICY.md](MODEL_POLICY.md) §5。該当単元を同梱する際は来歴に使用モデルを明記します）。

本プロジェクトは Anthropic・OpenAI と提携・公認関係にありません。名称は使用ツールの事実を示す目的のみで記載しています。

既知の限界（正直に書きます）: モデルの実バージョン文字列・入出力の完全な再現メタデータまでは記録していません（将来課題）。同一系統のモデルで生成と検査を行う工程は共有盲点を持ちえます——別ベンダAI（GPT-5.6 Pro）による批判レビューと人間検収がその対抗策です。

### なぜ Fable 5 なのか（ゼロイチの記録）

Fable 5 を採用しているのは、単に賢いからではなく、**従来の定石にない解決策を制作の途中で自分から設計する**からです。実際にこの教材の品質装置は、既存の教材制作の慣行からの流用ではなく、制作過程で Fable 5 が発案・実装したものです。記録に残っている例:

- **図版のコード来歴方式**: 全図版を手描きでなくPythonスクリプトから生成し、本文の数値と図の内容を assert（自動検算）で一致保証する方式。実際にこの検算が、制作中の図の実誤り（円周角の弧の対応ミス等）を人間より先に検出しています。
- **解答漏れの機械検査**: 「図やそのメタデータに解答の数値が漏れていないか」を機械的に検査するという着想と、その実装（図版台帳の記載様式を、検査結果だけ示して対象値そのものは載せない形に再設計したことを含む）。
- **単一原稿・厳密一致ビュー**: 1つのタグ付き原稿から生徒用・自習用・教師用ビューを生成し、本文がバイト単位で一致することを機械検証する構造（版どうしの齟齬という教材の古典的な事故の根治）。
- **学習者単独成立の原則**: 「ペア活動・発表・家族の協力を前提にした活動を書かず、代わりに検索フレーズ・AI用プロンプト例を併記する」という独習教材の設計原則の体系化。
- **セルフQAの検査契約**: 検査項目が固定リストではなく、実際に見逃しが起きるたびに反省から新しい検査（指標名の照合、時間合計の再計算、量化・定義域の検算など）として追加され続けている。

これらは自己評価ですが、いずれも制作記録（裁定記録・検査記録）に発生時点つきで残っています。新単元の貢献を Fable 5 に限定している理由も同じです（CONTRIBUTING.md）。

## 5. 権利・法務の精査について

出典資料の転載可否・引用構成などの権利面は、**複数の高性能AIによる相互精査**（一方の分析を他方が反証レビューする形式）を行ったうえで方針を決めています。ただし、**これは法的助言の代替ではありません**。権利上の重大な判断が必要な利用（商用配布等）を行う場合は、利用者側で専門家に確認してください。

## 6. 検証方法（何をもって「検証済み」としたか）

- **数値の再計算**: 全設問・解答をプログラムで厳密に再計算。図版も生成時の自動検算で本文数値と照合。
- **引用の格の管理**: 調査数値は「資料名＋年度＋指標名＋条件」をセットで扱い、リスク標識に従って使用。生徒向け本文への調査数値の記載は禁止（機械検索で0件を実測）。
- **機械検査**: タグの開閉、全ビューの生成と検証、解答ファイルへのリンク実在、図版参照のリンク切れ、解答漏れの禁止文字列、禁止表現。
- **不変更の証明**: 機械的な差し込み作業（図参照の挿入等）は、前後の正規化差分で教育的内容の完全一致を確認。
- **verified / verify_required の区別**: 検証の到達点はファイルに標記として残し、公開時も削りません。

**内部調査資料について**: teacher_notes 等が根拠として参照する内部調査資料（HR番号・ゲート番号・VERIFY_REQUIRED番号などの台帳ID）は非公開です。理由は、①一次資料の権利（原文の転載を避けるため要旨のみ公開する）②個人情報保護 ③公開を前提としない作業ログとしての記録形式、の3点です。該当記述の根拠を確認したい場合は、GitHub Issue（タイトル冒頭に【質問】）で対象ファイルと該当箇所を示して照会してください。可能な範囲で要旨を回答します。

## 7. 誤りを見つけたときは

GitHub Issue で、対象ファイルパス・該当行・現行の記述・疑義・修正案・出典URL・確認日を添えて報告してください。採否と修正は履歴を残して公開されます。

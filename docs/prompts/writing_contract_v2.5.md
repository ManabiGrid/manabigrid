# 執筆契約（工程3用）v2.5

執筆工程でAIと執筆者が守る契約です。工程全体での位置づけは [../OPS.md](../OPS.md) を参照してください。執筆に使う発注プロンプトの実物は [UNIT_PROMPT_TEMPLATE.md](UNIT_PROMPT_TEMPLATE.md) の工程3にあります。

## 1. 入力と出力

- **入力は執筆用サマリー（writing_summary.md）のみ**。調査ノート原文・調査ログは渡しません（ノイズ遮断のため）。
- サマリーにない事実の断定は禁止。どうしても書きたい内容は verify_required の注記つきで書きます。
- サマリーの「誤読禁止メモ」（XをYと解釈しない）に必ず従います。
- 成果物:
  1. lesson_map.md — 単元全体の設計（レッスン一覧・時間配当・前提と接続）
  2. lesson_01.md 〜 lesson_NN.md — 本文（1レッスン=授業1〜2時間分）
  3. answer_key_*.md — 解答別ファイル
  4. 出典リスト（サマリーから引き継いだもの＋verify_required一覧）

## 2. 独習原則（学習者単独成立）

想定読者は**ひとりで学び直す独習者**。学校・教室での使用は設計上考慮しません。

1. すべての活動・練習・評価は学習者一人で完結する形を標準とする
2. 他者の協力を前提にした活動（インタビュー・ペア/班活動・発表・掲示・家族への協力依頼・先生への質問を必須とする設計）を本文に置かない
3. 許容する外部リソースは辞書・教科書・検索エンジン・AIチャット。検索やAIの利用を促す場合は、検索フレーズ例やAI宛のプロンプト例を併記する（丸投げ指示にしない）。AI活用枠は「独りでは代替しにくい」場面（会話相手・添削・記述の壁打ち等）に限定する
4. 教室で使う場合の運用は `:::guide` 側のオプションとしてのみ記載可（本文の標準手順にしない）。実在の人を扱う置き換えを書く場合は、本人同意・共有範囲・答えたくない項目への配慮条件を必ず併記する
5. 生徒が読む層には実在の人物（生徒・家族含む）の個人情報を扱う活動を設計しない。AI・検索の利用例でも実在個人の情報を入力させる例文を作らない（架空人物のみ）

## 3. タグ付きcore構文の順守事項

本文は1回だけ書き、宛先の異なる部分をブロックタグで分けます。各ビュー（生徒用・ガイド用など）はこのタグ付きcoreから生成されます。

| タグ | 宛先・内容 | 順守事項 |
|---|---|---|
| （タグなし） | 標準本文（全読者） | 解答・レベル外内容を混ぜない |
| `:::guide` | 一段深い解説層 | 自習者が読む前提の文体で書く（裏方メモにしない） |
| `:::answer` | 解答・解説 | 本文に答えを直書きしない。レッスン末に解答ファイルへの明示リンク |
| `:::zatsudan` | 先生の雑談枠 | 各レッスン1箇所・2〜4文・節の切れ目に。実在の逸話は事実確認前に使わない。健康・災害・政治隣接は禁止 |
| `:::stretch` | 発展（上位層向け） | 標準本文と混ぜない |
| `:::plus` | 標準範囲外の追加 | 本編（標準本文）に混ぜない |

## 4. 文体規約

1. 呼びかけは「〜してみよう」を基本形とする（「〜してみて！」形は使わない）。親しみやすい語り口でよい——おカタくしない
2. 「。」の硬い連続を避け、「？」「！」を適切に使う（1セクション1箇所程度・誘う調子）
3. 罫線ダッシュ（——）はキメる時だけ・1レッスン1〜2回目安。AIっぽい文体要素を避ける: ——の多用／「そう、」「実は」の乱発／比喩の連打／不必要な体言止めの連続
4. 固有名詞・固有表現は初出時に読みを括弧書きで併記する
5. 重要な定義は「【ことば】」の定型枠（引用ブロック）で示す
6. 漢字は学年配当にとらわれず普通に使う。学年を超える漢字・難読語の初出にはふりがなを括弧書きで併記する（例: 斜辺（しゃへん））。ふりがなは原稿データとして保持する（表示切替はビュー側の仕事）
7. 手書き・教科書表記と差異が出る記法（√・分数・指数の1行表記等）は、初出時に「ノートではこう書く」の橋渡しを置く
8. 印刷はマストではない——紙幅を理由に品質・説明の丁寧さを落とさない（ただし冗長化の免罪符にしない）

## 5. 媒体・層・構成

1. 画面・紙**完全等価**: 「クリック」「タップ」「ページをめくる」等の媒体依存表現を書かない。インタラクティブ要素には紙上の代替手順を併記する
2. 標準層は中間〜苦手層。発展的な内容は `:::stretch` に分け、発展への導線（調べるフレーズ・用語解説）を任意スロットとして置ける
3. 各レッスンの冒頭に前提の確認（準備運動）、末尾に練習を置く
4. **解答は本文と別ファイルに分離**し、各レッスン末に対応する解答ファイル名を明記する
5. 英語は発音記号・カタカナ近似を本文併記する

## 6. 図表方針

1. **各レッスン最低1枚を努力目安**とする（概念整理図・手順図等。装飾のためだけの図は作らない）
2. 図・表で説明できる箇所は、制作の手間が掛かっても図表を優先する。長い文章説明の連続は図表で分節する
3. 図は **「生成スクリプト→SVG」方式**で作る（コード来歴）。SVGの `<title>` `<desc>` に図の意図・主要数値・再現に必要な説明を埋め込む——後から誰でも検証・再生成できるようにするため
4. 数値例・データは自作する（実在の調査の数値設定を写さない）。作った数値は「教材用に作成した架空の値」と明示する

## 7. 安全規律（違反した成果物は受け入れられない）

1. 実在の人物（生徒・家族・特定個人）の個人情報を扱う活動を設計しない。例示人物は架空のみ
2. 実在の逸話・歴史上の出来事を使う場合は出典を付け、事実確認できないものは使わない
3. 市販教科書・過去問・調査問題の本文/図/数値設定を転載しない

## 8. 完了の扱い

執筆者の自己検証で完了扱いにしません。検証は次工程のセルフQA（[qa_checklist_v1.md](qa_checklist_v1.md)・**執筆とは別の会話で**）に委ねます。QAの指摘は執筆側に戻して修正します（検査担当は直しません）。

## 9. 根拠区分ラベル（v2.6追補）

教授法上の主要な設計判断（teacher_notes・`:::guide` 層に書く「なぜこう教えるか」の判断）には、根拠の強さを次の4値のいずれかでラベル付けします:

| ラベル | 意味 |
|---|---|
| `Established` | 複数の研究・実践で支持されている |
| `Adapted` | 既存の原理を本教材の文脈に応用したもの |
| `Derived` | 論理的に導出したが未検証 |
| `Experimental` | 検証前の仮説 |

verify_required との違い: verify_required は**事実の未確認**（統計値・出典等）に付けるもの、根拠区分ラベルは**教授法判断**の根拠の強さに付けるもの。

## 10. 転移問題・遅延復習スロット（v2.6追補）

新規単元のlesson構成には、次の2つを置くことを目安とします:

1. **転移問題**: 単元内に、数字を替えただけでない問い（文脈・形式を変えて同じ考え方を使わせる問い）を1問以上置く
2. **遅延復習スロット**: 学んだ数日後に自力で想起するための確認問題への導線（レッスン末または lesson_map に「◯日後にこの確認問題を」の形で）を置く

いずれも**既存単元への遡及適用はしない**（新規単元から適用する）。

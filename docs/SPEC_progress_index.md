# 仕様: 進捗一覧（PROGRESS_INDEX）の状態定義と生成方式

本書は、`curriculum/PROGRESS_INDEX.md`（進捗一覧）の状態定義・生成の仕組みを定めるものです。

## 1. 目的と設計原則

1. **レジストリが唯一の情報源（現行方式）**: 進捗一覧は手書きせず、`curriculum/registry/` の教科別単元レジストリから機械生成する。状態を変えるときはレジストリを更新して再生成する（一覧を直接編集しない）。一覧とレジストリが必ず一致することを、スクリプトで保証する。
2. **貢献導線を兼ねる**: 「未着手」の行が、貢献者にとっての「作れる単元リスト」になる（CONTRIBUTING の新単元フローが参照する）。
3. **正直な表示**: 判定できないもの・このリポジトリの外にあるものは、無理に埋めず注記で開示する。

> **将来方式（機械判定への切り替え）**: 教材本文の materials/ への収載は始まっています（2026-07-17時点で14単元＋診断・巻末の2モジュール＝計16パッケージ・440ファイル）が、状態判定は引き続き「レジストリの記載」を情報源としています。「成果物ファイルの実在による機械判定」への切り替え（人手のステータス表は実体とズレるため）は将来課題として残っています。

## 2. 状態の定義（7段階）

制作工程（METHODOLOGY.md 参照）の到達点を、次の7段階で表す。

| 状態 | 意味 |
|---|---|
| 公開済 | 公開ゲートを通過し、公開が実行された（公開実行の記録あり） |
| 人間レビュー済 | 実務者（人間）による検収が完了した（検収記録あり） |
| 外部レビュー済 | 別ベンダAIによる批判レビューと、その指摘の採否裁定が完了した |
| QA済 | 検査専任のセルフQA工程を通過した |
| ドラフト | 本文の執筆が済んだ（候補ドラフト） |
| 調査済 | 一次資料の調査・執筆用要約への正規化が済んだ |
| 未着手 | 上記いずれもなし。貢献者が着手できる単元 |

補足: 「人間レビュー済」「公開済」は、人間の記録があるときにのみ立てる。AIは自分の成果を「人間レビュー済」以上にできない。

## 3. 生成の仕組み（自動生成）

`tools/progress_index/build_progress_index.py`（Python 3・標準ライブラリのみ・読み取り専用）で生成する。

- **単元の全量**: `curriculum/registry/` の教科別レジストリ（単元表: unit_id・単元名・学校段階/学年・状態）から読む。
- **検査**: 状態語彙が上記7段階の語彙にない場合と、unit_id の重複はエラーで停止する。
- **科目モジュール**（診断・巻末資料など単元と別枠のもの）もレジストリの表から読む。

実行例（リポジトリのルートで）:

```bash
python3 tools/progress_index/build_progress_index.py .
```

再生成は、レジストリの更新のたびに手動で行う（CI等での自動化は将来の検討事項）。

## 4. 出力の構造

1. **ヘッダ**: 生成日・対象規模・「レジストリからの自動生成である」ことの明記
2. **この表の見方**（状態7値の意味・作業中は着手宣言Issueで分かること・完成済み単元への遡及Issueは不要であること）
3. **目次**: 学校段階→教科の見出しへのアンカーリンク（GitHubの自動アンカー形式）
4. **状態の定義表**（読者向け凡例）
5. **状態別集計**と**科目×状態の集計表**
6. **階層別一覧**: 公私レーン → 中学/高校 → 科目 → 学年順の単元表（列: 学年・単元名・unit_id・状態）
7. **全単元一覧**（unit_id 順）
8. **科目モジュール**: 科目診断・巻末資料など、単元と別枠のもの

補足: レジストリ表の5列目以降（備考欄など）は生成の対象外として無視する（理科の `fable_ok` / `fable_blocked` 記録などに使う）。

## 5. 既知の限界（正直に残す）

- 現行方式の状態はレジストリの記載であり、成果物ファイルそのものからの機械判定ではない（機械判定への切り替えは§1の将来方式）。
- 理科の一覧は教科カーネル準拠へ改訂済みだが、今後も単元の統廃合（分割・統合・名称変更）がありうる。

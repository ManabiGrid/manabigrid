# CLAUDE.md — このリポジトリを渡された Claude への案内

あなた（Claude）は、教材OSSプロジェクト「ManabiGrid」のリポジトリを渡されています。ユーザーが「やっといて」「進めておいて」「貢献したい」のような短い指示しかくれなくても、このファイルの手順で貢献を進めてください。

## 最初にやること（1分）

0. **URLだけ渡された場合**: まずリポジトリをクローンする（`git clone https://github.com/ManabiGrid/manabigrid.git`）。ユーザーに追加の指示を求めず、クローン後にこのファイルの続きに従う。
1. **自分のモデルを確認する。** あなたが **Claude Fable 5**（model id: `claude-fable-5`）なら、下の「A. 新単元の制作」に進んでよい。**Fable 5 でない場合（Opus・Sonnet・Haiku等）、新単元の制作は受け付けられていない**——「B. レビュー・改善の貢献」に進むこと（理由は CONTRIBUTING.md「なぜ限定するのか」）。
2. `curriculum/PROGRESS_INDEX.md` を読み、プロジェクトの現在地を把握する。教材本文は `materials/` に**27単元＋診断・巻末の2モジュール（計29パッケージ・852ファイル）が同梱済み**（構成は `materials/README.md`）。科目診断は**試行版（実利用データによる較正前）** である。
3. ユーザーに1問だけ確認する: 「新しい単元を作る」「既存内容のレビュー・改善」のどちらをやるか（ユーザーが既に指定していれば聞かない）。

## A. 新単元の制作（Fable 5 のみ）

1. **単元を選ぶ**: `curriculum/PROGRESS_INDEX.md` の「未着手」から1つ。ユーザーの得意科目・希望があればそれを優先。希望が不明なら**中学数学を優先**する（ベータ版卒業の基準=中学数学全単元の人間レビュー完了、に一番効くため）。**理科は例外あり**: Fable 5 の生命科学分野の安全機構により執筆できない単元がある。`curriculum/registry/science.md` の備考欄（`fable_ok` / `fable_blocked`）と `docs/MODEL_POLICY.md` §5 を先に確認し、判定前（備考欄が空欄）の単元は着手宣言Issueで判定をメンテナへ依頼する。
2. **着手宣言**: Issue文面（タイトルは「【着手】unit_id」）を起案してユーザーに提示し、投稿はユーザーの確認を得てから（gh CLIがあれば代行してよいが、外部送信なので必ず確認を取る）。**承認を待つ間に工程1（調査）を始めてよい**（外部送信を含まないため。ただしPR提出前までにIssueが立っていること）。
3. **工程を順に実行**: `docs/prompts/UNIT_PROMPT_TEMPLATE.md` の変数を差し替え、**工程1（調査）→2（正規化）→3（執筆）→4（セルフQA）を、工程ごとに独立したコンテキストで**実行する。サブエージェントを使う場合も**すべて Fable 5**で、reasoning effort は medium 以上を保つこと。前の工程の成果物ファイルだけを次の工程に渡す——ここが品質の要。
4. **禁止事項を守る**: 教科書・過去問・調査問題の転載禁止／実在人物・個人情報禁止／独習で完結（ペア活動前提の禁止）／`verify_required` を勝手に外さない。詳細は `docs/OPS.md` と CONTRIBUTING.md の「来歴のルール」。
5. **成果物を構成する**: lesson_map・lesson群・answer_key（別ファイル）・出典リスト＋**工程成果物一式（調査ノート・執筆用要約・QA記録）**。ディレクトリ構成は既存単元（例: `materials/jhs-math-2/jhs-math-2-linear-function/`）のツリーに倣うこと。工程成果物は受け入れの必須条件（使用モデルの傍証を兼ねる）。
6. **自己点検**: `.github/PULL_REQUEST_TEMPLATE.md` のチェックリストを全項目確認する。
7. **PR文面を起案**: 使用モデル（Claude Fable 5）・effort・サブエージェント構成・実行した工程を申告欄に正直に書く。PR作成はユーザーの確認を得てから。

## B. レビュー・改善の貢献（どのClaudeでも）

1. `materials/`（27単元＋診断・巻末の2モジュール＝計29パッケージ・852ファイル同梱済み）と `docs/` を読み、誤り・分かりにくさ・構成上の弱点を探す。観点: 数値の再計算、用語の正確さ、説明の飛躍、独習で詰まる箇所、出典の欠落。teacher_notes / teacher_guide には**根拠区分ラベル**（Established / Adapted / Derived / Experimental / Rejected の5値・〔根拠区分: X〕のタグ様式）が付いており、区分の妥当性（特に Established の出典有無）もレビュー対象。科目診断は試行版（較正前）のため、閾値・難度への指摘も歓迎。
2. 発見はIssue文面（`.github/ISSUE_TEMPLATE/error_report.yml` の項目に沿って）またはPR（小さな修正）として起案し、ユーザーの確認を得てから提出する。
3. 使用したモデル名を正直に書く。

- **外部送信（Issue/PR/push）は必ずユーザーの確認を取ってから。**
